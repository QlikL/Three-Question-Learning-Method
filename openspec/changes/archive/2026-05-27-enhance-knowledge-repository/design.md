## Context

当前复合知识库模块缺少完整的文件展示和交互功能。用户需要一个现代化的知识库界面来查看和管理AI推荐的学习资料。项目采用模块化架构，使用原生JavaScript、IndexedDB存储和Canvas渲染。

**当前状态：**
- 已有学习空间（知识图谱、争议点、测评、资料推荐）
- 已有测评中心和个人中心
- 复合知识库模块存在但功能不完整
- 使用阿里云百炼大模型生成内容

**约束：**
- 纯前端项目，无后端服务
- 使用IndexedDB进行本地数据存储
- 需要支持在线文档链接和本地文件两种类型
- 保持现有UI风格一致性

## Goals / Non-Goals

**Goals:**
- 实现完整的文件列表展示功能
- 支持在线文档链接跳转和本地文件下载
- 提供文件搜索和分类筛选功能
- 实现文件元数据展示（标题、来源、时间、大小）
- 保持与现有UI风格一致
- 支持IndexedDB持久化存储

**Non-Goals:**
- 不实现文件上传功能（由AI生成推荐）
- 不实现文件内容编辑功能
- 不实现文件共享或协作功能
- 不实现离线文件缓存（仅存储元数据）

## Decisions

### 1. 文件存储方案
**决策：** 使用IndexedDB存储文件元数据，不存储文件内容
**理由：** 
- 纯前端项目，文件内容可能较大
- 在线文档只需存储URL
- 本地文件由用户自行下载管理
**替代方案：** localStorage（容量限制）、后端存储（增加复杂度）

### 2. 文件类型识别
**决策：** 基于URL扩展名和MIME类型进行文件分类
**分类：**
- `pdf`: PDF文档
- `doc/docx`: Word文档
- `xls/xlsx`: Excel表格
- `ppt/pptx`: PowerPoint演示
- `link`: 网页链接
- `other`: 其他类型

### 3. UI组件设计
**决策：** 创建独立的FileList组件，采用卡片式布局
**理由：**
- 复用性高，可独立测试
- 卡片式布局适合展示文件元数据
- 符合现有组件架构模式

### 4. 文件预览方式
**决策：** 
- 在线文档：新标签页打开链接
- 本地文件：触发浏览器下载
**理由：** 纯前端无法直接预览文件内容，需依赖浏览器能力

### 5. 搜索和筛选实现
**决策：** 前端内存搜索，不依赖后端
**理由：**
- 文件数量有限（通常<100个）
- 避免增加后端复杂度
- 响应速度快

## Risks / Trade-offs

**[Risk] 在线链接可能失效** → **Mitigation:** 显示最后验证时间，提供链接报错反馈机制

**[Risk] 文件元数据不准确** → **Mitigation:** AI生成时提供置信度，用户可手动编辑元数据

**[Risk] 大量文件导致性能问题** → **Mitigation:** 实现虚拟滚动或分页，限制单次加载数量

**[Trade-off] 纯前端搜索 vs 后端搜索** → 选择前端搜索，牺牲大规模搜索能力换取简洁性

**[Trade-off] 不缓存文件内容** → 节省存储空间，但用户需重复下载

## Migration Plan

1. **Phase 1:** 创建FileList组件和基础样式
2. **Phase 2:** 实现IndexedDB文件存储结构
3. **Phase 3:** 集成AI文件推荐功能
4. **Phase 4:** 实现搜索和筛选功能
5. **Phase 5:** 测试和优化

**Rollback:** 保留原有知识库页面，新功能通过特性开关控制
