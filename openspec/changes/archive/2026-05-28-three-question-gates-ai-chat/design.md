## Context

三问学习法应用是一个纯前端HTML/JS/CSS项目，使用Vanilla JS + IndexedDB + localStorage实现。当前三问流程（第一问心智模型、第二问根本性分歧、第三问深度测试）已基本实现，但存在以下问题：

1. **无通过条件**：用户可以跳过任何学习步骤直接进入下一问，学习质量无法保证
2. **知识图谱布局**：使用简易环形布局，节点容易重叠，连线交叉多
3. **缺乏AI交互入口**：心智模型的"第一性原理"和根本性分歧模块缺少"去问AI"的快捷入口

**现有架构关键组件：**
- `LearningPage` - 学习页面主控制器，管理三问流程
- `MentalModelList` - 心智模型展示组件，支持展开/折叠
- `DebatePanel` - 争议点对比面板，左右分栏展示
- `QuizItem` - 测评题目组件，单选/判断题
- `KnowledgeGraphRenderer` - Canvas知识图谱渲染
- `AiChatWidget` - AI聊天悬浮组件

## Goals / Non-Goals

**Goals:**
- 为三问添加可验证的通过条件门控
- 改善知识图谱的布局质量，消除节点重叠
- 在心智模型和根本性分歧中添加上下文感知的AI交互入口
- 保持现有功能完整，新功能作为增量添加

**Non-Goals:**
- 不改变三问的内容生成逻辑（AI提示词不变）
- 不修改数据模型结构（IndexedDB存储结构不变）
- 不改变知识图谱的节点内容和数据格式
- 不实现新的AI模型调用，复用现有的aiService

## Decisions

### 决策1：门控状态追踪方式

**选择：组件内部状态追踪 + 回调通知**

在 `MentalModelList` 和 `DebatePanel` 组件内部维护展开/查看状态，通过回调函数通知 `LearningPage` 更新门控条件。

**备选方案：**
- 方案A（排除）：在LearningPage中通过DOM查询判断展开状态 - 耦合度高，依赖组件内部DOM结构
- 方案B（排除）：在Store中持久化展开状态 - 过度设计，展开状态是临时性的

**理由：** 组件内部状态追踪最符合现有架构模式，MentalModelList已有展开/折叠逻辑，只需添加状态计数和回调。

### 决策2：第一问门控逻辑

**选择：监测5个心智模型标签全部展开**

在MentalModelList中添加 `_expandedSet` 追踪已展开的模型ID，当集合大小等于5时通过回调通知LearningPage。LearningPage在收到通知后启用"进入第二问"按钮。

**理由：** 使用Set数据结构天然去重，避免重复计数。展开即视为"查看"，与心智模型的展开/折叠交互逻辑一致。

### 决策3：第二问门控逻辑

**选择：监测3个根本性分歧全部被切换查看**

在DebatePanel中添加 `_viewedTopics` Set追踪已查看的主题索引，初始时当前主题自动标记为已查看。当集合大小等于3时通过回调通知LearningPage。

**理由：** 用户通过点击不同分歧标签来查看，每次点击即视为"已查看"。初始显示的主题也计入。

### 决策4：第三问通过条件

**选择：答题完成后判定正确率≥90%**

在QuizItem的 `_submitAnswer` 方法中，当所有题目都已作答时，计算正确题数。只有正确数≥9时触发onComplete回调，否则显示未通过提示并允许重试。

**备选方案：**
- 方案A（排除）：每答一题即时判定 - 无法支持"全部作答后判定"的需求
- 方案B（排除）：答错即中断 - 过于严格，用户体验差

**理由：** 用户需要完成全部10题后才能看到是否通过，符合"作答完十条题目之后并且正确题数达到9题"的需求。

### 决策5：知识图谱布局优化

**选择：改进力导向布局 + 迭代碰撞检测**

在现有 `_layoutNodes` 方法基础上：
1. 增加节点间最小距离计算（基于节点半径 + 固定间距）
2. 多轮迭代碰撞检测推开重叠节点
3. 添加连线交叉减少的引力/斥力调整
4. 增大Canvas尺寸以提供更多布局空间

**备选方案：**
- 方案A（排除）：使用D3.js force layout - 引入外部依赖不符合项目纯JS风格
- 方案B（排除）：使用预定义树形布局 - 不适用于知识图谱的网状结构

**理由：** 在现有Canvas绘制框架内优化，不引入新依赖。多轮迭代可有效消除重叠，引力/斥力调整可减少连线交叉。

### 决策6：AI交互入口实现

**选择：扩展AiChatWidget支持openWithMessage方法**

在AiChatWidget中添加 `openWithMessage(message)` 公共方法，该方法：
1. 打开聊天窗口
2. 自动将message填入输入框并发送

心智模型和根本性分歧组件中的"去问AI"按钮调用此方法。

**备选方案：**
- 方案A（排除）：每个按钮独立创建聊天弹窗 - 重复代码，与现有聊天组件不统一
- 方案B（排除）：使用window.open打开新页面 - 用户体验割裂

**理由：** 复用现有AiChatWidget组件，保持一致的聊天体验。openWithMessage是最小侵入式的扩展方式。

## Risks / Trade-offs

- **[门控过严]** 用户可能觉得必须展开所有标签过于繁琐 → 缓解：展开操作本身是学习的一部分，且展开后不会自动折叠
- **[知识图谱性能]** 多轮迭代碰撞检测在节点较多时可能影响渲染性能 → 缓解：限制迭代次数为5轮，节点数通常不超过20个
- **[AI调用成本]** "去问AI"按钮可能增加API调用频率 → 缓解：使用现有的aiService，已有防抖和上下文管理
- **[答题重试]** 第三问未通过时需要支持重试 → 缓解：提供重试按钮，清空答题记录重新作答
